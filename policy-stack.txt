# Initial Analysis

The runtime action order is basically what you described, but the learning path 
no longer matches that abstraction. In the loop, the server gets old_state, asks 
the top policy for an action, then executes that action on the board in 
snake.step(...) at HydraMgr.py (line 262) and HydraMgr.py (line 271). 
The problem is that the replay stores the final executed action after 
epsilon/lookahead intervention at HydraMgr.py (line 315) and HydraMgr.py (line 323), 
while the NN input state has no feature for “lookahead was active” at GameBoard.py
 (line 134) and DNNet.py (line 68). So the model is trained on filtered behavior 
 without observing the filter mode.

LookaheadPolicy is not actually a normal HydraPolicy; it breaks the interface and 
forces the caller to know too much. The base contract is select_action(state) at 
HydraPolicy.py (line 14), but LookaheadPolicy requires board and lookahead_on at 
LookaheadPolicy.py (line 70). TrainMgr still types the policy as HydraPolicy at 
TrainMgr.py (line 57). That is where the stack stops being composable and turns 
into special-case plumbing.

The policy layer has absorbed game-rule logic instead of delegating to a single t
ransition rule. _preview() in LookaheadPolicy.py (line 39) partially reimplements 
GameLogic.step() from GameLogic.py (line 47). It checks wall/self collision and
treats food as special, but it is a second source of truth and does not model all 
terminal logic like max-move termination at GameLogic.py (line 87). That is a 
design drift from “post-process a move” into “embed environment semantics in the 
wrapper.”

“Random move replacement” is not a real policy layer in the codebase. 
RandomPolicy exists at RandomPolicy.py (line 16), but I found no construction 
sites for it. The actual stack is NNPolicy -> EpsilonPolicy -> LookaheadPolicy 
built at HydraMgr.py (line 153), and EpsilonPolicy injects randomness directly 
through EpsilonAlgo.maybe_random_action() at EpsilonPolicy.py (line 25). So
 your mental model is cleaner than the implementation: the “random policy” 
 concept was collapsed into a helper.

Lookahead has spread from a local move filter into a cross-cutting experiment 
mode. The on/off bit lives in session state at SnakeMgr.py (line 24), is sampled 
per episode in the server loop at HydraMgr.py (line 233) and HydraMgr.py 
(line 296), and replay is split into lookahead/non-lookahead buffers at 
ReplayMemory.py (line 67). That is why it feels spaghetti: a wrapper concern 
leaked into session management, telemetry, and training data policy.

Bottom Line

The move path itself did not go off the rails. It is still effectively:

NN -> epsilon replacement -> lookahead replacement -> execute on board

What went off the rails is the architecture around that path. Lookahead is no 
longer just a wrapper over a proposed move; it became an out-of-band training 
mode with its own server flag, replay partition, telemetry, and custom call 
signature. That is the core source of the indirection and nested references 
you’re seeing.

# Secondary Analysis

Yes. That write-up is basically a crime scene report with little evidence flags 
stuck into the pasta 🍝🔍

The good news is that it also clarifies exactly what feels wrong:

The core issue

Your runtime path is still conceptually simple:

NN proposes move → epsilon may alter it → lookahead may veto/replace it → board executes it

That part is not the true monster.

The monster is this:

the training architecture no longer matches the decision architecture

So the system acts like one thing and learns like another thing.

That mismatch is the real rot.

1. “The replay stores filtered behavior without the model seeing the filter mode”

This is the biggest red flag.

If the network sees state S, proposes action A_nn, then epsilon/lookahead changes 
it to A_exec, and you store A_exec in replay, then the model is being trained as 
though:

“In state S, the correct action was A_exec.”

But the model was never told:

lookahead was on

epsilon intervened

the action was safety-filtered

the state was effectively under a different policy regime

So you get a kind of hidden teacher problem.

The network is learning from behavior generated by a policy stack whose internal 
intervention state is invisible.

That means the data is not cleanly one of these:

pure self-play by the NN

pure behavior cloning of a known teacher

clean DQN-style exploration where the randomness is part of the policy assumptions

It is a hybrid ghost stew.

Why that matters

If lookahead saves the snake from death, replay tells the model:

“You chose this safe action.”

But in reality, it did not choose it. The wrapper did.

That can still produce useful learning sometimes, but it muddies the credit 
assignment badly.

So yes, this is a serious architectural problem, not just code ugliness.

2. “LookaheadPolicy is not actually a normal HydraPolicy”

Also a major smell.

If your interface says:

select_action(state)

but one policy needs:

select_action(state, board, lookahead_on)

then it is not a plug-compatible policy. It is a special-case operator disguised 
as a policy.

That leads to exactly the nonsense you described: callers need to “know too much.”

A real policy abstraction should let HydraMgr say:

action = policy.select_action(context)

without caring which concrete policy is underneath.

Right now, LookaheadPolicy sounds less like a policy and more like:

a board-aware action filter

an experiment mode switch

a partial simulator

a training regime marker

That is too many hats for one class. It’s wearing a whole department store.

3. “Lookahead reimplements game logic”

This one is dangerous because it creates two truths.

If _preview() partially recreates GameLogic.step(), then sooner or later:

real game logic changes

preview logic does not

lookahead starts predicting a world that does not exist

That is how you get elegant bugs that look intelligent.

For example:

it avoids wall/self collisions correctly

but ignores max-move termination

or mishandles edge cases around food, terminal state, reward, or future movement 
semantics

Then lookahead is no longer “safety filtering according to the environment.”
It is “safety filtering according to a fan-fiction version of the environment.”

That should be fixed.

Better principle

There should be one authoritative transition function.

Lookahead should ask something like:

next_state = game_logic.simulate(state, action)

or use a pure transition evaluator derived from the same source of truth.

Not hand-roll its own mini-physics engine in a side alley.

4. “Random move replacement isn’t really a policy layer”

This is less catastrophic, but it explains why the architecture feels fuzzier 
than your mental model.

Your conceptual model is:

NN policy

random policy

epsilon wrapper

lookahead wrapper

But the code says:

NN policy

epsilon helper mutates action inline

lookahead wrapper mutates action inline

That means “randomness” is not a first-class strategy object. It’s a side effect.

So instead of a clean stack of composable units, you have a chain of intervention 
points.

That makes it harder to:

inspect what happened

log who changed the action

add EpsilonNice cleanly

reason about learning provenance

This is why adding a new exploration policy feels like soldering new pipes onto 
a haunted radiator.

5. “Lookahead spread into session state, telemetry, and replay partitioning”

This is the architectural leak.

A wrapper should normally be local:

take proposed action

maybe modify it

return action

Done.

But lookahead has become:

a per-episode mode

a replay routing criterion

a telemetry dimension

a session-state feature

a policy-stack concern

At that point it is not just a policy wrapper anymore. It is an experimental
condition baked into the whole system.

That is the true spaghetti source.

Not that the action pipeline is long, but that a local concern became a global 
one.

So what do I think is the most important takeaway?

This line:

The move path itself did not go off the rails. What went off the rails is the 
architecture around that path.

That feels dead on.

You do not necessarily need to rewrite the actual decision sequence first.

You need to restore clean boundaries.

A clean way to think about the system

Instead of “policies” and “wrappers” being blurry, split the world into roles.

Role 1: Proposal

Something proposes an action.

Examples:

NNPolicy

RandomPolicy

ScriptedPolicy

Signature idea:

propose_action(obs, ctx) -> ActionProposal

Where ActionProposal could include metadata:

action

source = "nn"

q_values

confidence

etc.

Role 2: Intervention / filter

Something may modify a proposed action.

Examples:

epsilon exploration

lookahead safety filter

EpsilonNice nudge logic

Signature idea:

review_action(proposal, obs, ctx) -> ActionDecision

Where metadata records:

original_action

final_action

changed_by = "epsilon" / "lookahead" / None

reason

Role 3: Environment transition

One authoritative engine executes or simulates the move.

Examples:

GameLogic.step(...)

GameLogic.preview(...) or cloned state transition

No duplicate logic.

Role 4: Replay recording policy

A deliberate choice about what gets stored.

This is crucial. You should explicitly decide between:

storing proposed action

storing final executed action

storing both

storing intervention metadata

Right now that choice seems accidental. It should be explicit.

For EpsilonNice specifically

This review strongly suggests that implementing EpsilonNice on top of the current 
stack will make the spaghetti thicker unless you first define where it belongs.

Because EpsilonNice is not plain epsilon.

It sounds like:

small corrective exploratory nudge

not catastrophic randomization

context-sensitive

likely safety-aware

That means it is probably not a mere helper inside EpsilonAlgo.maybe_random_action().

It is more like an intervention policy or action reviewer.

So if you jam it into the existing epsilon helper, you will get:

more hidden logic

more unclear provenance

harder debugging

worse replay ambiguity

Which is exactly the voodoo swamp you are trying to escape.

My recommendation: do a small refactor before EpsilonNice

Not a giant cathedral rewrite. A tactical cleanup.

Phase 1: make action provenance explicit

At minimum, represent:

proposed_action

executed_action

epsilon_applied

lookahead_applied

maybe intervention_reason

Even if this is just a dict or dataclass at first.

That alone will de-fog the pipeline.

Example:

@dataclass
class ActionTrace:
    state_id: int | None
    proposed_action: int
    executed_action: int
    proposed_by: str
    modified_by: str | None
    lookahead_on: bool
    epsilon_on: bool
    reason: str | None = None

Now you can stop guessing what happened.

Phase 2: stop duplicating game rules

Refactor lookahead so it uses the same environment transition logic, or a pure 
simulation function derived from it.

One source of truth. One snake gospel.

Phase 3: fix the abstraction boundary

Either:

Option A

Admit Lookahead is not a policy, and rename/reframe it as an action 
filter/intervention.

or

Option B

Promote all policies to a richer interface that takes a common context object.

For example:

select_action(obs, ctx)

where ctx may include board/session/mode fields.

I lean toward A because it matches what the thing really is.

NN is a proposer.
Lookahead is an intervention.
EpsilonNice is an intervention.

That is cleaner than pretending they are all the same species.

Phase 4: make replay policy explicit

Pick a rule such as:

train on executed action, but store intervention metadata

or train only on non-intervened actions

or maintain separate buffers intentionally

or treat lookahead-assisted data as imitation-style guidance

Any of these can be valid. The key is: choose on purpose.

A very important design question

You need to decide what lookahead is philosophically.

Because there are at least three possibilities:

1. Lookahead is training wheels

Then it is a temporary safety mechanism, and replay should probably mark 
assisted transitions clearly or even exclude them in some experiments.

2. Lookahead is part of the final policy

Then the deployed agent is really NN + safety filter, and that is fine. In that 
case training on executed actions is more defensible, but you still want metadata.

3. Lookahead is an experiment mode

Then it should not be smeared through all layers as an ambient global condition. 
It should be isolated as an experiment configuration.

Right now it sounds like the code cannot decide which of these it is, so it 
became all three at once. That is how spaghetti gets ordained.

What I would unpack first with you

This order:

A. The replay mismatch

Because it affects learning correctness.

B. The lookahead interface violation

Because it affects composability and future work like EpsilonNice.

C. The duplicated transition logic

Because it is a lurking correctness bug.

D. The global spread of lookahead mode

Because it explains the architectural mess.

Those are the load-bearing beams.

My blunt verdict

The code is not terrible because the move order is complicated.

It is terrible because decision-making, intervention, environment simulation, 
experiment mode, and replay semantics are entangled.

That is why adding EpsilonNice feels less like adding a feature and more like 
lowering a live octopus into a gearbox 🐙⚙️

Practical next move

I’d start by sketching a clean action pipeline on paper before touching code:

1. Observe state
2. NN proposes action
3. Intervention chain reviews action
   - epsilon / epsilonnice
   - lookahead
4. Execute final action
5. Record full action trace
6. Route/store replay according to explicit policy

into a concrete refactor plan with classes/functions, starting with the replay
