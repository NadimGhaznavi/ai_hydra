self._history in RNNPolicy.py (line 19) was meant to give the RNN actual temporal context at inference time.

The reason is in training: RNNTrainer.py (line 77) builds tensors shaped like [B, T, F], where T is a sequence of consecutive game states. So the RNN is trained to learn from a window of states, not from one isolated state. But the old policy was passing only a single state vector each step. RNNModel.forward() (line 42) allows that by turning a 1-D input into [1, 1, F], which means the recurrent part only sees one timestep. In practice that makes the RNN behave almost like a plain feed-forward model at inference.

self._history is just a rolling buffer of the last DRNN.SEQ_LENGTH states. On every move, the policy appends the newest state and sends the whole window into the model as [T, F], which RNNModel.forward() converts to [1, T, F]. That lets the RNN use the same kind of sequential signal at inference that it saw during training.

reset_episode() clears that buffer so one snake run does not bleed into the next. Without that, the first states of a new episode would be contaminated by the tail end of the previous episode.

The important point is that _history was not for turbo mode specifically. It was there to fix a training/inference mismatch in the RNN path. Turbo likely just made that mismatch more obvious.

A cleaner alternative would be to keep the RNN hidden state directly between calls instead of replaying a history window. Your current RNNModel.py (line 48) throws the hidden state away (x, _ = self.m_rnn(x)), so _history was the simpler way to preserve temporal information without redesigning the model API.
